# Decisions

- Use a TypeScript control-repo scaffold with `src/`, `tests/`, `docs/`, and `.github/workflows/` as the long-lived layout for future provisioning/auth/policy work.
- Keep operator-facing provisioning input centered on `repo_slug` and `description`; expose `execution_mode` only as an internal validation/testing mode with `dry-run` default.
- Reject uppercase and underscores rather than silently rewriting them; normalization is limited to trimming inputs and deriving the final `proj-<slug>` repository name.
- Split the GitHub boundary into `src/github/auth.ts` (JWT + installation-token exchange/cache), `src/github/client.ts` (REST wrapper methods), and `src/github/permissions.ts` (shared permission matrix for docs/tests/runtime validation).
- Require the auth layer to validate one superset GitHub App repository permission matrix up front so later provisioning, hardening, and requester-review tasks can assume the installation token is authorized for their shared boundary calls.
- Use exactly one approved template source identified by `PROVISIONING_TEMPLATE_REPOSITORY=<owner>/<repo>` with default ref contract `main`; no alternate templates or per-request template selection are supported.
- Persist requester identity as canonical lowercase `REQUESTER_LOGIN` and mirror it into `.github/provisioning/requester-metadata.json` with schema `{ kind, schema_version, requester_login, provisioned_at, provisioned_by_workflow }`; if the variable and file are both available they must match exactly or consumers fail closed.
- Keep Task 4 orchestration on one linear stage model: validate request, resolve execution mode, resolve the approved template source, preflight duplicates, then either emit a planned create (`dry-run`) or perform a private create-from-template (`sandbox`).
- Define duplicate repos as an idempotent preflight failure: an existing `proj-*` target stops the workflow before any create call and is reported as `duplicate_target_preflight: failure`, never as a reusable or partial-success state.
- Use classic branch protection, not rulesets, as the only baseline hardening mechanism for `main`; encode one canonical payload with `required_status_checks.strict=true`, `requester-review-policy` required, `required_approving_review_count=1`, empty push restriction allowlists, and `enforce_admins=true` so there is no admin/direct-push bypass path in the classic model.
- Adopt an explicit provisioning outcome model: `success | failed | quarantined | not_ready` with top-level `readiness` and `scopeSuccess` flags so intermediate failures never masquerade as success.
- Define quarantine as mandatory when repository creation succeeds but hardening application/verification fails; include machine-readable remediation payload in both stage output and top-level workflow result.
- Treat requester-review enforcement as a readiness gate represented by `enforcement_readiness_verify`; until enforcement verification is available, return `not_ready` instead of `success` for full-readiness claims.
- Count requester-review approvals only from the latest review per reviewer, require the review to target the current PR head SHA, ignore requester self-approval when requester authored the PR, and model "authorized reviewer" via collaborator permission levels `admin|maintain|write` so non-counting approvals are deterministically excluded.
- Mark enforcement readiness ready when the `requester-review-policy` workflow file exists and the canonical evaluator module is available; keep real merge-behavior proof deferred to Task 8 sandbox/e2e verification.
- Count Task 8 "generated evidence" only when it is written under `.sisyphus/evidence/` by the harness itself, includes machine-readable JSON for the scenario result, and is explicitly labeled `LOCAL_SIMULATION_ONLY` when live GitHub behavior was not exercised.
- Use one rerunnable harness (`npm run evidence:task8`) plus scenario-specific entrypoints for success and requester-policy-failure paths so final reviewers can reproduce exact artifacts without manual interpretation.
- Treat requester metadata persistence as part of sandbox provisioning success criteria: fail/quarantine when either `REQUESTER_LOGIN` upsert or `.github/provisioning/requester-metadata.json` commit fails after repository creation.
- Promote enforcement readiness to a target-repository evidence check (workflow file + metadata file + requester variable observed through GitHub API) instead of a local control-repo file heuristic.
- Standardize all GitHub App runtime secret names on `GITHUB_APP_ID`, `GITHUB_APP_INSTALLATION_ID`, and `GITHUB_APP_PRIVATE_KEY` across workflows, runners, and operator docs.
- Finalized Task 8 operator-facing cleanup: refreshed requester-metadata contract example to `.github/workflows/provision-repository.yml@...`, and regenerated all checked-in Task 8 evidence artifacts via `npm run evidence:task8` so JSON/TXT/manifest now reflect current orchestration shape (including `template_artifact_verify` stage and `templateArtifactsVerified` scope).
